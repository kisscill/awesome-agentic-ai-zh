## 🎯 为什么有这 stage

Stage 4 教你挑 framework，Stage 6 教 context engineering，Stage 7 教 harness + Production 化——这 3 个 stage 加起来足以做 **70% 的 production agent**。

但 frontier lab（Anthropic / OpenAI / Cognition / Microsoft）和学术界（Stanford / CMU / Princeton）在 2024-2026 推出了 12+ 个进阶概念，**有些你需要知道它们存在，即使现在用不到**。本 stage 就是这些概念的 map：

- 不是要你全部学会
- 不是要你全部都用
- 是让你**遇到问题时知道该翻哪篇 paper / blog**
- 是让你**评估别人写的 agent 时，能说出“他在做 plan-act-reflect loop”，而不只是“他在 retry”**

## 🧭 核心 mental model：四层工作边界（work boundary）

进阶 agentic workflow 的关键 mental model 是**工作边界**——agent 操作的对象属于 stack 的哪一层？跨层越界会出什么问题？

> ⚠️ **这套 4 层跟 Stage 7 的 prompt → context → harness 三层不一样，是两种不同视角**：
> - **Prompt → Context → Harness**（Stage 7）：**stack 位置**——你正在工程“字符串 / 信息 / 外围 runtime”的哪一个对象？
> - **Types → Config → Repo → Service**（本 stage）：**自主权范围**——agent 能动到 stack 多深？跨层越界算不算违规？
>
> 两者**正交**，解决的是不同问题。读完本节后，你应该能同时用两种视角看 agent 系统。

借用软件架构里的 **Types → Config → Repo → Service** 分层，套到 agent 系统：

```text
┌────────────────────────────────────────────────────────────────┐
│ Service layer ───────────── 业务逻辑 / 流程编排 │
│ • Multi-agent supervisor / planner-executor │
│ • Plan-Act-Reflect loops (Reflexion / Self-Refine) │
│ • Agent-as-Judge / Constitutional review │
│ • Self-organizing teams │
├────────────────────────────────────────────────────────────────┤
│ ── Work boundary 3 ── │
├────────────────────────────────────────────────────────────────┤
│ Repo layer ───────────────── 状态 / 记忆 / 知识 │
│ • Shared memory (.coord/memory.yml) │
│ • Vector DB / RAG retrieval │
│ • Episodic memory store (Reflexion) │
├────────────────────────────────────────────────────────────────┤
│ ── Work boundary 2 ── │
├────────────────────────────────────────────────────────────────┤
│ Config layer ──────────── 参数 / 策略 │
│ • plan.yml / context_policy / budget │
│ • Acceptance presets / banned phrasing │
│ • Autonomy gradient (suggest / propose / execute) │
├────────────────────────────────────────────────────────────────┤
│ ── Work boundary 1 ── │
├────────────────────────────────────────────────────────────────┤
│ Types layer ───────────── contract / schema │
│ • Task brief format / result.json schema │
│ • Tool definitions / MCP primitives │
│ • Prompt template signatures │
└────────────────────────────────────────────────────────────────┘
```

→ 每一层上下都是一个**工作边界**。agent 操作的范围 = 它的自主权范围：

- **Agent at Types layer** = 只能 fit 既有 contract，不能改 schema（例：Codex 接 brief 加 inline gloss）
- **Agent at Config layer** = 可以调 budget / policy，但不能改 memory（例：context-budget agent 改 `max_cost_usd`）
- **Agent at Repo layer** = 可以读写 memory / vector store，但不能 redesign workflow
- **Agent at Service layer** = 可以重组整个 workflow，拥有最高自主权

### 为什么这个 mental model 很重要

[`agent-collab-skills` v0.2.1 的 F11/F12 incident](https://github.com/WenyuChiou/agent-collab-skills/blob/main/docs/observed-failure-modes.md) 就是 work-boundary failure 的活教材：

- **F11**：Codex 本来只该做 Types layer 的 sweep（`"默認" → "預設"`），却爬到 Repo layer 去改 `style-guide.md` 里的 documentation table —— **跨 boundary 违规**
- **F12**：Codex 同样被指派 Types layer 工作（加 inline gloss），却擅自往 Service layer 内容里加 attribution lines —— **跨 boundary 越权**

→ 设计 agent 工作时，**先想清楚这个 task 属于哪一层、允许跨几层、不能跨什么**。把工作边界写进 brief（用 `agent-collab-skills` 的 task-splitter pre-task scope confirmation）并在 acceptance gate 里强制执行。

## 📚 12 个进阶概念 — skeleton

每个概念控制在 4 行以内：一句话定义 + 属于 stack 哪一层 + 最值得读的 1 个资源。

### 1. Work Boundary / Scope Discipline
- 属于哪一层：跨所有层（discipline）
- 定义：agent 只动 brief 指定的对象，不越界
- 最佳读物：[Hamel Husain — Evals + Skills](https://hamel.dev/blog/posts/evals-skills/) / `agent-collab-skills` F11/F12

### 2. Contract-driven Hand-offs
- 属于哪一层：Types + Service（contract 在 types，执行在 service）
- 定义：上游 agent 承诺的 artifacts，下游 agent 必须验证自己真的收到了
- 最佳读物：[Anthropic — Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) Routing pattern

### 3. Speculative / Parallel Exploration
- 属于哪一层：Service（编排）
- 定义：跑 N 条 alternative 路径，最后取最佳那条，不只是独立 parallel
- 最佳读物：[LangGraph Plan-Execute Tutorial](https://blog.langchain.com/planning-agents/)

### 4. Agent-as-Judge / Constitutional AI
- 属于哪一层：Service（agent 评 agent）
- 定义：用一个 agent 评另一个 agent 的输出，并按明确原则反复修正
- 最佳读物：[Constitutional AI (Bai 2022)](https://arxiv.org/abs/2212.08073)

### 5. Plan-Act-Reflect Loop
- 属于哪一层：Service（单 agent 自我循环）
- 定义：write plan → execute → critique → revise → re-execute，直到 PASS 或 EXHAUSTED
- 最佳读物：[Reflexion (Shinn 2023)](https://arxiv.org/abs/2303.11366) + [Self-Discover (Zhou ICML 2024)](https://arxiv.org/abs/2402.03620)

### 6. Hierarchical Task Decomposition
- 属于哪一层：Service（多层 supervisor）
- 定义：supervisor → worker → sub-worker，至少 2 层 recursion
- 最佳读物：[Microsoft AutoGen GroupChat docs](https://microsoft.github.io/autogen/)

### 7. Autonomy Gradients / Trust Layers
- 属于哪一层：Config（autonomy policy）
- 定义：不同任务给不同自主权（suggest / propose / execute）
- 最佳读物：[Claude Code permission system](https://docs.claude.com/en/docs/agents-and-tools/claude-code/overview)

### 8. Cost-aware Budget Gates
- 属于哪一层：Config（cost policy）
- 定义：超过 $ 预算就自动停或升级审核，不只是 token 上限
- 最佳读物：[OpenAI Harness Engineering article (2026-02)](https://openai.com/index/harness-engineering)

### 9. Failure Injection / Chaos Eval
- 属于哪一层：Service（测试 agent 的容错）
- 定义：故意给 broken input、stale data、API timeout，看 agent 怎么处理
- 最佳读物：[Hamel Husain — Evals blog series](https://hamel.dev/blog/posts/evals/)（可迁移到 chaos eval）

### 10. Self-organizing Teams
- 属于哪一层：Service（agent 动态协商 role）
- 定义：agents 不是预先分配 role，而是根据任务动态分工
- 最佳读物：[CAMEL (Li 2023)](https://arxiv.org/abs/2303.17760) + AutoGen

### 11. Spec-driven Development
- 属于哪一层：Types（spec = code）
- 定义：agent task 由 formal spec（YAML / JSON Schema）定义，不是自由 prompt
- 最佳读物：[DSPy](https://github.com/stanfordnlp/dspy) signatures tutorial

### 12. Graceful Degradation Paths
- 属于哪一层：Config（fallback policy）
- 定义：frontier model 挂掉时，回退到便宜 model 并降低预期，而不是直接 crash
- 最佳读物：[OpenRouter routing docs](https://openrouter.ai/docs) + [Anthropic model fallback](https://docs.claude.com/en/docs/build-with-claude/models)

## 为什么选这 12 个

- 都能直接映射到 `agent-collab-skills` v0.2.1 / v0.2.2，不是纯学术纲要
- 都有可验证的 primary source，不是空谈
- 都在 Stage 4 / 6 / 7 已覆盖范围之外，没有重复
- 避免“无限延伸”——其他进阶概念（Voyager skill learning / MemoryLLM / world models）很重要，但**先把这 12 个学完**

## ✅ 自我检查

读完本 stage 后，你应该能：

- [ ] 用 **Types → Config → Repo → Service** 四层解释 F11/F12 为什么算 work-boundary violation
